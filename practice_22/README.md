# Практическое занятие №22 — Балансировка нагрузки в веб-приложениях

## Описание

В данной практической работе реализована система балансировки нагрузки для веб-приложения с использованием **Nginx** и **HAProxy**. Запущены несколько backend-серверов на разных портах, настроены алгоритмы балансировки и механизмы отказоустойчивости.

## Технологии

- **Node.js + Express** — backend-серверы
- **Nginx** — основной балансировщик нагрузки
- **HAProxy** — альтернативный балансировщик

## Структура проекта

```
practice_22/
├── backend/

│   └── package.json
├── nginx.conf         # Конфигурация Nginx-балансировщика
├── haproxy.cfg        # Конфигурация HAProxy
└── README.md
```

## Backend-сервер

Простой Express-сервер, который возвращает свой идентификатор и порт. Идентификатор задаётся через переменную окружения `SERVER_ID`.

Запуск трёх экземпляров:
```bash
cd backend && npm install

# Терминал 1 — Сервер 1 (порт 3000)
PORT=3000 SERVER_ID=1 node server.js

# Терминал 2 — Сервер 2 (порт 3001)
PORT=3001 SERVER_ID=2 node server.js

# Терминал 3 — Резервный сервер (порт 3002)
PORT=3002 SERVER_ID=3 node server.js
```

## Настройка Nginx

### Алгоритмы балансировки

| Алгоритм         | Описание                                              |
|------------------|-------------------------------------------------------|
| Round Robin      | По умолчанию. Запросы распределяются по кругу         |
| Least Connections| Запрос → сервер с минимальным числом соединений        |
| IP Hash          | Клиент всегда попадает на один и тот же сервер         |

### Запуск Nginx

```bash
# Проверка конфигурации
sudo nginx -t -c $(pwd)/nginx.conf

# Запуск
sudo nginx -c $(pwd)/nginx.conf

# Перезагрузка без остановки
sudo nginx -s reload

# Остановка
sudo nginx -s stop
```

### Тестирование балансировки

```bash
# Многократные запросы — будут приходить ответы от разных серверов
for i in $(seq 1 6); do curl -s http://localhost/ | jq .server; done
```

### Настройки отказоустойчивости

В `nginx.conf` настроены:
- `max_fails=2` — сервер считается недоступным после 2 неудачных попыток
- `fail_timeout=30s` — недоступный сервер исключается на 30 секунд
- `backup` — резервный сервер подключается только при недоступности основных

## Настройка HAProxy

HAProxy запускается на порту `8080`, статистика доступна на порту `8404`.

### Запуск HAProxy

```bash
# Ubuntu/Debian
sudo apt install haproxy

# Запуск с конфигурацией
haproxy -f haproxy.cfg

# Или как системный сервис
sudo cp haproxy.cfg /etc/haproxy/haproxy.cfg
sudo systemctl restart haproxy
```

### Тестирование HAProxy

```bash
for i in $(seq 1 6); do curl -s http://localhost:8080/ | jq .server; done
```

Статистика HAProxy: `http://localhost:8404/stats`

## Сравнение Nginx vs HAProxy

| Критерий          | Nginx                          | HAProxy                          |
|-------------------|--------------------------------|----------------------------------|
| Основное назначение | Веб-сервер + балансировщик   | Специализированный балансировщик |
| Статистика        | Нет встроенной                 | Встроенная панель на /stats      |
| Производительность | Высокая                       | Очень высокая                    |
| SSL Termination   | Да                             | Да                               |
| Health Checks     | Пассивные (в community)        | Активные (TCP/HTTP)              |

## Ключевые моменты реализации

- Nginx использует блок `upstream` для описания группы backend-серверов
- HAProxy разделяет конфигурацию на `frontend` (приём запросов) и `backend` (серверы)
- Параметр `check` в HAProxy активирует активные проверки доступности серверов
- Резервный сервер (`backup`) подключается автоматически при недоступности основных
